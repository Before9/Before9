Restful

发表于 2019-01-03

Restful：

一种软件架构风格、设计风格，而不是标准，只是提供了一组设计原则和约束条件。它主要用于客户端和服务器交互类的软件。基于这个风格设计的软件可以更简洁，更有层次，更易于实现缓存等机制。

这么理解试试看，也就是你听过建筑的复古风格，欧美风格之类的么，大体就是一个意思。如果你盖个别墅不按设计风格走，直接以小茅屋的形式，没有属于任何风格它也能盖成不是，这就是我们平时开发API不按RESTFUL风格也可以是一个道理。REST 指的是一组架构约束条件和原则，满足这些约束条件和原则的应用程序或设计就是RESTful。

Restful发展历程：

详细发展历程：https://blog.csdn.net/a78270528/article/details/78469758
Restful风格==面向过程—>面向资源—>面向标签—>面向文档

面向过程==想起一出是一出，只要有功能就写新接口

面向资源==封装单个接口，添加一个接口，删除一个接口，修改一个接口，多重逻辑需要多个if-else

面向标签==封装单个接口，通过method进行语义化操作 post/put/delete/get

面向文档==具备面向标签的功能语义化，在返回响应的时候告诉客户还可以做什么操作

生活场景

面向过程：每次都写新接口，就比如说我们去星巴克喝咖啡，这个时候我们想要点一杯咖啡，点一杯咖啡需要写一个接口，下完单之后我突然想到今天要早点睡不能喝咖啡，我要取消订单，那这个时候就需要再写一个接口，每一次下单，取消订单，修改订单都要重新写接口

面向资源：封装单个接口，还是星巴克的例子，老板意识到每次都写新接口不行，然后这个老板就招了几个员工，分工明确，下单一个接口，取消订单一个接口，修改订单一个接口，但是这样用户不知道哪个是下单的，哪个是取消订单的，就要写好多个if-else,会浪费用户的时间，拖慢这个程序

面向标签：这个时候老板就想了，那我直接把他们挂牌分开咋样，就是我们的通过method进行语义化操作，类就是星巴克的大门，每个函数方法都是一个小牌，下单是post,取消订单是delete,修改订单是put,这样用户就方便了很多。

面向文档：挂小牌的方法很好，但是有的用户不仔细看不知道还可以有别的操作，老板就召集员工开了个小会，通知他们用户去每个小牌前的时候员工顺便告诉一下用户还可以有别的操作， 比如有用户的朋友临时不来了，但是这个用户点了两杯，那这个时候员工就需要告诉他您的另一杯饮品是可以取消的，这就是restful完美的境界了

  • 说这么多，到底restful是什么？

    接口规范

  1. 统一资源接口
    RESTful架构应该遵循统一接口原则，统一接口包含了一组受限的预定义的操作，不论什么样的资源，都是通过使用相同的接口进行资源的访问。接口应该使用标准的HTTP方法如GET，PUT和POST，并遵循这些方法的语义。

  2. URL中只使用名词来指定资源，原则上不使用动词

  3. 用HTTP协议里的动词来实现资源的添加，修改，删除等操作。即通过HTTP动词来实现资源的状态扭转
    GET 用来获取资源，
    POST 用来新建资源（也可以用于更新资源），
    PUT 用来更新资源，
    DELETE 用来删除资源。

  4. Server和Client之间传递某资源的一个表现形式，比如用JSON，XML传输文本，或者用JPG，WebP传输图片等。当然还可以压缩HTTP传输时的数据（on-wire data compression）。

  5. 用 HTTP Status Code传递Server的状态信息。比如最常用的 200 表示成功，500 表示Server内部错误等。

一些其他规范：

规则1：URI结尾不应包含（/）
规则2：正斜杠分隔符（/）必须用来指示层级关系
规则3：应使用连字符（ - ）来提高URI的可读性
规则4：不得在URI中使用下划线（_）
规则5：URI路径中全都使用小写字母
